# Proposal: observe-enrichment — the Layer B signals the coordination views need

**Status:** Draft — proposed 2026-07-17. The five design decisions (below) and the all-four-signals / phased-cheapest-first scope were settled with the owner on 2026-07-17; pending review of the write-up and the companion ADR (ADR-0008).

## Context

The three coordination views shipped their **Layer A** skeleton — the parts buildable from the existing model: the cross-change DAG, waves and coarse per-target status, Category/Type badges, gate/approval state, emergency + freeze, and the two-layer graph edges ([coordination-ui-views.md](coordination-ui-views.md) §"Layer A — buildable now"). What remains is **Layer B**: four richer per-stage signals SCP does **not** capture today. This proposal designs their observe-signal shape, docs-first, before implementation.

The gap is structural, not a rendering oversight. SCP has **two executor read verbs feeding two paths**, and neither persists these signals:

- **`observe()`** — a self-rescheduling pg-boss loop (`OBSERVE_QUEUE`, ~60s cadence, worker-only), deliberately slower than the 1s reconcile tick to respect external-API rate limits (`apps/server/src/coordination/observe.ts:154-213`). It flattens each `ExecutorEvent` into a `change_source_events` row and consumes **only the `correlation` hints** for matching; the full provider payload it carries in `event.raw` is stored but never parsed downstream (`observe.ts:67-113`). The argocd plugin's `observe()` reads `GET /api/v1/applications` and holds the entire `ArgoApplication` — including `status.health.status` and `status.sync.revision` — verbatim in `event.raw`, yet emits only a bare `sync` event and populates neither `commitSha` nor `artifactDigest` (`packages/plugins/argocd/src/index.ts:230-255`). **So health and revision arrive on the observe path and are dropped.**
- **`status(ref)`** — called every 1s reconcile tick, but **only for already-coordinating wave targets, never for standing health**. It already computes the rich signals: `mapArgoPhase` reads `sync.status` / `health.status` (Healthy/Progressing/Degraded/…) / `operationState.phase` / `sync.revision` and returns `ExecutionStatus{phase, detail:`sync=X health=Y op=Z`, stateRef=sync.revision, progress}` (`packages/plugins/argocd/src/index.ts:134-224`). But reconcile's success path discards it: `updateWaveTargetObserved` persists only `{status, lastObservedAt, updatedAt}` (`apps/server/src/coordination/reconcile.ts:593-622`; `apps/server/src/coordination/wave-targets-repo.ts:155-165`). Only on fail/abort does `detail` survive, into a block Decision.

At the schema level, `change_wave_targets` has **no column** for observed version/digest/health/rollout-state; its only jsonb blobs are `executorRef` (an `ExternalRunRef {externalId, url?}` run handle, **not** a version) and `priorStateRef` (a pre-trigger rollback snapshot, **not** a live observation) (`apps/server/src/db/schema.ts:602-633`; `packages/plugin-api/src/index.ts:125-128`). And `ExecutionStatus` itself is a coarse 5-value phase enum + opaque `stateRef` + best-effort `progress`, with no field for digest, canary weight, analysis verdict, or health (`packages/plugin-api/src/index.ts:130-139`).

**The coordinate-not-execute boundary governs the shape.** Principle 1 and DESIGN §11/§12 make the boundary structural: the `ExecutorPlugin` interface deliberately has no `execute()`/`deploy()` verb (`DESIGN.md:534-540`), and there is a credential asymmetry. Every Layer B signal is therefore **observed/mirrored** via `observe()`/`status()` — SCP records progressive-delivery state, it never drives a canary step.

## The four signals

### (1) Per-stage version / image digest — `moderate-new` (version half is near-free)
- **Source:** `status().stateRef` is **already** `status.sync.revision` (the synced git commit SHA) — computed every observing tick, then discarded (`packages/plugins/argocd/src/index.ts:218-223`; `reconcile.ts:593-622`). `priorStateRef` proves the plumbing already round-trips this revision into a persisted column (`reconcile.ts:800-816`), so persisting the **live** revision is near-zero new plumbing. The **image digest** is genuinely new: the plugin reads only `sync.revision`, never the deployed images — it needs a plugin-side read of the Application resource-tree / managed-resources.
- **Storage:** additive nullable jsonb `observed_state` on `change_wave_targets`, mirroring the `executorRef`/`priorStateRef` jsonb precedent on the same row; written by extending `updateWaveTargetObserved` to take an observed-state payload (`wave-targets-repo.ts:155-165`); surfaced through `toChangeWaveTargetShape` (`plan-service.ts:146-162`) and an additive field on `ChangeWaveTargetSchema` (`status` is already `z.string()`, so additive with no wire break — `packages/schemas/src/changes.ts:154-173`). **No new table.**

### (2) Argo Rollouts progressive-delivery state — `heavy-new`
- **Source:** not observed at all today. The argocd plugin models only the `ArgoApplication` CR (sync/health/operationState) and never touches the `argoproj.io` Rollout CR or `AnalysisRun` (`packages/plugins/argocd/src/index.ts:134-148`). The clean observe path is to extend `observe()`/`status()` to read the Application resource-tree (`GET /api/v1/applications/{name}/resource-tree` or `/managed-resources`), where the embedded Rollout CR and its step/canary/analysis status surface — staying inside the ArgoCD API SCP already integrates (no new credential, no new executor, purely a read = a charter-clean mirror).
- **Storage:** the same jsonb `observed_state`, carrying a structured rollout sub-state (step index, canary weight, analysis phase/result, promote/abort). Requires a **parallel additive widening** of the executor-facing `ExecutionStatus`, which today truncates everything to `phase|detail|stateRef|progress` and has no rollout vocabulary (`packages/plugin-api/src/index.ts:130-139`). DESIGN §18 already frames deeper canary analysis as an additive extension (`DESIGN.md:763`).

### (3) Gate verdicts with reasons — `cheap-existing`
- **Source:** already captured and API-surfaced — this is **internal explainability, not an executor observe signal**. The gate verdict + full per-policy/per-effect `reasonTree` live in the `decisions` table (`schema.ts:464-480`; `evaluate.ts:255-271`), exposed at `/decisions/{id}` + `/decisions?subjectId=` (`routes/changes.ts:422-445`). Control-run outcomes (status + evidence + detail + `decisionId`) are at `/changes/:id/control-runs` (`routes/governance.ts:100-138`). Freezes are folded into the block `reasonTree` and CRUD-surfaced (`gate-orchestrator.ts:288-299`; `routes/governance.ts:336-441`). Approval quorum counts come from `quorumStatus` via `/approvals` (`approvals-repo.ts:177-188`).
- **Storage:** reuse the existing Control-outcome + Decision machinery (`DESIGN.md:485-514`; `control_runs` at `schema.ts:681-700`). The views **read** these onto the promotion arrows; they invent no verdict store. The **only** genuinely new piece is a **typed managed-scan verdict**: the `ControlPlugin` allowlist is `webhook-control` only (`control-runner.ts:14-18`), Trivy exists strictly as observe-only bundled infra (`BUILD_AND_TEST.md:407`), and control `status` is a fixed 6-value enum with untyped evidence/detail (`controls-repo.ts:90-99`). Treat that scan-verdict shape as an adjacent track (see Open decisions).

### (4) Component / service health — `heavy-new`
- **Source:** no poll-less health path exists. `observe()` only emits change-detection events (reconciledAt advanced); argocd health is read only while a wave target is actively coordinating, then discarded on success (`observe.ts:239-254`). There is no `HealthSourcePlugin`, no Prometheus plugin, and `PluginKind` has no health-provider kind (`plugin-api/src/index.ts:73-99`). The metric is **owner-provided**: either an owner push API (annotation record) or a health-source binding that polls a Prometheus query / HTTP probe on the existing 60s observe cadence.
- **Storage:** an **object-scoped observe/annotation signal** keyed to the component/service graph object — **not** a bespoke health table and **not** `objects.properties` (which would bump the object revision every scrape and store no history; objects carry only free-form properties/labels today — `packages/schemas/src/graph.ts:84-104`). Charter principle 2 and DESIGN §4.1's "thin projection tables that reference their graph object" (`DESIGN.md:213-217`) sanction a projection record referencing `objects(id)`. Health/canary are already enumerated charter **Operational Control** categories (`PROJECT_CHARTER.md:1865-1872`), so this is expressing the metric over the existing model, not a bespoke health-monitoring subsystem.

## Incremental plan (cheapest / highest-value first)

1. **Surface existing gate verdicts + reasons (signal 3).** Read `decisions.reasonTree` + `control_runs` + freezes + approval quorum onto the promotion arrows (open / blocked-on-scan-or-window / awaiting-approval, with the *why*). Zero new modeling.
2. **Persist the live synced revision (signal 1, cheap half).** Add the jsonb `observed_state` column; extend `updateWaveTargetObserved` to store `status().stateRef` — the revision reconcile already computes and discards. Powers the version staircase near-free.
3. **Capture the image digest (signal 1, new half).** Add a plugin-side read of the Application resource-tree / managed-resources for deployed image digests; populate `observed_state`.
4. **Argo Rollouts progressive-delivery state (signal 2).** Widen `ExecutionStatus` with an optional typed rollout sub-state; extend argocd `observe()`/`status()` to read the resource-tree Rollout CR; store into `observed_state`. **Mirror-only, never driven.**
5. **Owner-provided health metric (signal 4).** Define the ingestion seam (owner push API vs health-source binding) + the object-scoped observation record, then the optional graph health-dot overlay.
6. *(Adjacent track)* A typed managed-scan-verdict `ControlPlugin` (Trivy/OpenSCAP), so a scan verdict becomes a first-class `control_run` rather than a webhook-relayed opaque blob — tracks with the managed-scanning direction, independent of the above.

## Charter alignment

- **Coordination, not execution (1):** every signal is read-only observation via `observe()`/`status()`; no `execute()`/`deploy()` primitive is added (`DESIGN.md:534-540`). Rollouts canary/analysis/promote/abort is **mirrored**, never orchestrated.
- **Graph-native, no new top-level table (2):** per-target signals land as additive columns / a jsonb `observed_state` on the existing `change_wave_targets` projection row (DESIGN §4.1 "hot lifecycle state"); health lands as an object-scoped projection record referencing `objects(id)`; gate verdicts reuse `decisions`/`control_runs`.
- **API-first, additive within /v1 (3):** `status` is already `z.string()` and the read schema absorbs a new jsonb field without a break; new read projections (per-service board, release-chain, health overlay) are additive endpoints the UI consumes only via the generated SDK.
- **Explainability (6):** promotion blocks already carry a `decision_id`; the arrows surface it, they don't invent it.
- **Simplicity (7):** reuse existing mechanisms — the discarded `status()` richness, the `decisions` table, the executor-binding/observe pattern — rather than adding engines.

## Decisions (settled with owner, 2026-07-17)

1. **Health ingestion — owner push API now, binding-ready.** An owner push API writes an object-scoped health observation record; the record is shaped so a later **health-source binding** (a Prometheus query / HTTP probe polling on the existing 60s observe cadence, mirroring the executor-binding / source-mapping pattern) can write the **same** record without a schema change. SCP does not poll app health itself for MVP — this keeps the coordinate-not-execute posture and matches "optional owner-supplied metric"; the polling binding is deferred until an org actually needs SCP-side sampling.
2. **Enriched observed-state storage — a single additive nullable jsonb `observed_state` column on `change_wave_targets`** (the last-observed snapshot), mirroring the `executorRef`/`priorStateRef` jsonb precedent on the same row. Keeps the current-state read the views need to a single-row lookup. **Not** a per-observation time-series table (add one only if progression *history* is ever required) and **not** `object.properties`.
3. **Progressive-delivery detail — mirror READ-ONLY state only** (current step index, canary set-weight %, analysis phase/result, promote/abort/paused flag) via the ArgoCD Application resource-tree; **add NO promote/abort/retry verb.** The `ExecutorPlugin` interface keeps its no-execute-primitive shape (DESIGN §11/§12); the ADR carries a normative restatement of observe-not-drive.
4. **`ExecutionStatus` — a new optional, typed `observed` field**, not an overload of the free-form `detail` string or the rollback-reserved `stateRef`; keeps version/digest/rollout signals machine-readable for the views.
5. **Typed managed-scan verdict — a separate, parallel track.** The gate-verdict surfacing (signal 3) reuses the existing `decisions`/`control_runs` machinery and needs no scan plugin (an org running its scanner behind a `webhook-control` binding already produces a `control_run` today); a native Trivy/OpenSCAP `ControlPlugin` tracks with the managed-scanning direction — noted here as adjacent so the views can render a typed verdict later, but this proposal does not depend on it.

**Scope (owner-decided):** the proposal covers **all four** signals; implementation lands **incrementally in the cheapest-value-first order** above — each step its own PR — starting with gate-reason surfacing and the near-free synced-revision persist.

## Companion ADR

**An ADR is warranted.** The [coordination-ui-views proposal itself calls for one at Phase 4/5](coordination-ui-views.md) because these phases add genuinely new observe/annotation surface — not mere surfacing — and CLAUDE.md's convention is that significant post-approval decisions get an ADR, with precedent in the sibling ADR-0006 (fail-closed status) and ADR-0007 (Type taxonomy), both wave-target modeling decisions. Scope the ADR to the genuine model forks: **(a)** the enriched-observe storage shape (additive jsonb `observed_state` on `change_wave_targets` vs `object.properties`, no new top-level table); **(b)** the new object-scoped health/annotation signal type; **(c)** a normative restatement that Rollouts state is observed/mirrored, never driven. Signal 3 (gate-verdict surfacing) largely reuses `decisions`/`controls` and does **not** need ADR coverage. Worth a small DESIGN.md §14 (UI) / §9 note pointing at the observe-enrichment; this proposal is the umbrella and per-phase PRs reference it.

## Non-goals / deferred

- **SCP running progressive delivery.** Canary/analysis/promote happen inside Argo Rollouts; SCP mirrors them (charter: coordinate, not execute).
- **A bespoke health-monitoring subsystem.** Health is an optional owner-supplied up/down metric, not SCP polling app health.
- **A per-observation time-series of stage state.** The views render current per-stage state; history is deferred to a projection table if ever needed.
