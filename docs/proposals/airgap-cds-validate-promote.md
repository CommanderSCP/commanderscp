## Why now: E6 is universal, and the boundary walk is manual

**M17.3 E6 hard-refuses any cross-boundary export lacking passing, digest-bound scan evidence.** The export gate (`evaluatePromotionScanGate`, `apps/server/src/federation/promotion-repo.ts:116-140`, called from `exportPromotionBundle`) is deliberately universal and fail-closed: *"a MISSING scan refuses exactly like a FAILED one"* (`promotion-repo.ts:110-111`), **whether or not a scan-requirement policy is bound**. This is by design — scanning exists to authorize boundary crossing ([ADR-0013](../adr/0013-supply-chain-scan-sbom-manifest.md)), and E6 is the structural backstop [ADR-0018](../adr/0018-domain-local-dev-pipelines.md)'s dev exemption leans on. We are not proposing to weaken it by one bit.

But it has a hard consequence: **an org WITHOUT a pipeline scanner cannot promote into (or out of) an air-gapped domain at all.** Every export of a substantive artifact refuses at E6, because nothing in the org's pipelines ever produced the evidence. Today the only evidence ingress is the `scan-result-control` `ControlPlugin`, which **pulls** Trivy result JSON from a per-binding, operator-configured URL (`packages/plugins/scan-result-control/src/index.ts:1-8,43-46`) — evidence some external scanner produced. No scanner in the pipeline, no evidence, no crossing. For the pipeline-less orgs the Managed Execution Exception exists to serve, the air-gap story currently ends at a `409`.

And even for orgs that **can** produce evidence, the CDS crossing itself is a six-step manual operator walk (runbook `docs/runbooks/retrans-relay.md` on the M15.5(c) branch, PR #112):

1. **Commander:** `scp federation promote --out` **twice** — one `.scpbundle` addressed to the retrans, one addressed to the destination outpost.
2. **Operator:** walk the retrans-addressed `.scpbundle` to the retrans instance.
3. **Retrans:** `scp federation import` (M17.4(a) verifies the manifest), then `scp federation relay` (skopeo pull → validate → signed tarball into `SCP_RELAY_OUT_DIR`).
4. **Operator:** carry the tarball **and** the outpost-addressed `.scpbundle` across the CDS.
5. **Operator:** distribute the retrans instance's cosign public key out-of-band.
6. **Outpost:** `scp federation import` of the `.scpbundle`, then `scp federation relay-import --file … --change … --pubkey …`.

Only the M17.4(b) pre-deploy byte verify is automatic. Everything upstream of it is a human typing commands and walking files.

**M13 completes the air-gap story for both populations:**

- **§13.3 (managed scanning)** makes scan-and-sign a **first-class step of the commander's promotion process** (owner decision 2026-07-23): the commander scans the artifacts under promotion (scanners selected per artifact type from the scanner registry), evaluates the results against the M17.5 requirements, and only then signs and exports — so E6's evidence exists **by construction** for every promotion, pipeline or no pipeline. Org-pipeline scan evidence remains a supported alternate ingress through the existing shapes, with **zero gate-code changes**.
- **§13.1 (the staging node) + §13.2 (DeliveryTarget)** automate the boundary walk — steps 2–4 and 6 become an unattended loop; the operator's remaining job is whatever their CDS product requires of them.

Everything rides machinery that exists: the M15.5(c) relay pipeline and its 409 role gate (PR #112, `apps/server/src/federation/retrans-relay.ts`), the caller-independent import verify paths (`import-repo.ts`, `promotion-repo.ts`, `importRelayTarball`), the pg-boss self-rescheduling tick pattern (`apps/server/src/coordination/observe.ts::startObserveLoop`), `ScanEvidenceSchema`'s designed-for-a-second-scanner shape (`packages/schemas/src/supply-chain.ts:180-181`), the `scp-managed-iac` runner pattern (DESIGN.md §12), and the digest-pinned tool-vendoring discipline (`tools/cosign/pin.env`, `tools/skopeo/pin.env`, #111).

## 13.1 The staging node — the retrans role, formalized

The role already exists. [ADR-0004](../adr/0004-service-naming-commander-outpost-retrans.md) defined `retrans` as a CDS-boundary store-and-forward validation relay — *never originates config, holds no local authoritative objects, never terminates a promotion* — and shipped it declared, not built. M15.5(c) (PR #112) activated its arm: `buildRelayTarball` hard-refuses `409` unless `self.role === "retrans"` (`retrans-relay.ts:356-360`), while destination import runs on any role. What remains manual is the *walk*: someone has to import what arrived, run the relay, and drop what leaves.

**The staging node is that walk, automated.** A `role: retrans` instance runs an unattended loop — and the same inbox loop, running on the receiving outpost, automates the destination end (step 6 of the walk):

```
inbound arrival (inbox) ──▶ auto-import ──▶ auto-relay build ──▶ outbound drop (DeliveryTarget, §13.2)
```

- **Inbound arrival.** An **inbox** — a filesystem directory by default, an S3 prefix under §13.2 — where the two channel artifacts land: `.scpbundle` metadata bundles and `scp-relay-*.tar.gz` byte tarballs, deposited by the low side's DeliveryTarget drop or by the org's CDS on the high side.
- **Auto-import — role-aware.** The loop lists the inbox and routes each file to the **existing** verify path for its kind; which paths apply depends on where the loop runs. **At the retrans:** `.scpbundle` → the federation import (checksum + Ed25519 exporter-signature verify, hash-chained journal replay, foreign-origin rejection in `import-repo.ts`; promotion bundles additionally get the M17.4(a) manifest-signature + set-equality verify in `promotion-repo.ts`); a relay tarball arriving at a retrans (the high-side hop) is **validated and forwarded, never imported** — tarball signature + per-file checksums + authorized-set cross-check, then the onward drop — because the retrans has no registry to push into (deployment profile, below). **At the receiving outpost** (destination import runs on any role, as today): `.scpbundle` → the federation import; relay tarballs → `importRelayTarball` (tarball signature, per-file checksums, local-authorized-set cross-check, push-by-digest into the outpost's local Gitea + re-inspect). **Zero changes to the verification semantics**: import verification is caller-independent today — the CLI names a file and POSTs; the loop names a file and calls the same repos. The only structural change is on the retrans tarball hop: `importRelayTarball` runs its signature/checksum/authorized-set checks inline with the registry-push flow (it resolves the dest-push credential up front), so the retrans's push-less validate-and-forward extracts those same checks to run without the push half — a refactor, not a new trust decision. Everything else automates only *who names the file*.
- **Auto-relay.** When a promotion import succeeds on a `retrans`-role instance, the loop schedules `buildRelayTarball` for it. The 409 role gate stays exactly as PR #112 built it — the staging node satisfies it because it *is* the retrans; commanders and outposts still cannot relay.
- **Outbound drop.** The built tarball and the onward outpost-addressed `.scpbundle` are written via the destination peer's DeliveryTarget (§13.2). Filesystem default = today's `SCP_RELAY_OUT_DIR` behavior, unchanged.

**Trigger.** A pg-boss self-rescheduling tick loop cloned from `startObserveLoop` (`apps/server/src/coordination/observe.ts` — the established singleton pattern: `boss.work` handler re-`send`s with `startAfter` + `singletonKey`), scanning the inbox on an interval (`SCP_STAGING_TICK_INTERVAL_SECONDS`, default 60). That is the **default and the backstop**. Where poke-mode is enabled, the [ADR-0009](../adr/0009-optional-poke-mode-federation.md) poke chain is the low-latency option: the commander pokes the reachable low-side retrans → immediate cycle; the high-side retrans pokes the outpost after dropping (ADR-0009's hop-by-hop propagation, owner-clarified 2026-07-18). The poke never replaces the tick — a dropped poke self-heals on the next interval, exactly the ADR-0009 sparse-safety-net model.

### Deployment profile (owner decision 2026-07-23)

**The staging node is NOT an outpost minus features — it is a purpose-built relay.** The owner decided its shape (2026-07-23): the retrans is deliberately very simple. Its whole job is (1) **validate the cosign signature** — the **transitive proof of scan-pass** (scans execute at the commander before signing, §13.3; an artifact only carries a valid commander signature if it passed scans): the exporter-signed promotion manifest plus the per-artifact signatures, verified by the same M17.4 machinery every import runs; (2) **place the artifact into the CDS** via the destination peer's DeliveryTarget (§13.2), whether that is an S3 bucket, a drop directory, or whatever else the org's CDS takes; (3) **record the pass-through**. It ships none of the other things outposts come with. Concretely:

- **One codebase, slim profile.** The same SCP server binary/image every instance runs, started in a `role: retrans` profile with every other subsystem off: **no local Gitea/registry stack** ([ADR-0010](../adr/0010-outpost-local-artifact-infra.md) outpost local infra never applies at the boundary — the M15.4 role matrix in `tools/helm-verify` already allots `retrans` **zero** bundled backends), **no executor coordination, no deploy machinery, no coordination UI**. Not a second binary, not a new deployment artifact — a profile of the one server, which is what keeps the verify code paths byte-identical to everywhere else they run.
- **Tools: cosign AND skopeo — and why both.** The owner's "probably just cosign" is the verification half, and it is correct about verification: cosign verifies the manifest and per-artifact signatures but **moves no bytes**. skopeo (vendored digest-pinned, #111) is what actually pulls the OCI bytes from the source registry and materializes the OCI-layout tarball — without it the relay could attest and relay nothing. Verify with cosign, move with skopeo; the profile needs both already-vendored tools and no others (no Trivy, no `oscap`, no `tofu` — see §13.3 on why the staging node never scans).
- **State: the same PostgreSQL, exercising only the relay footprint.** The owner's "small database to track what's been passed through" is a small *slice* of the same database, not a new store (principle 4 — Postgres stays the only required stateful dependency): federation peer rows with their cosign public keys, the `bundle_transfers` pass-through ledger (the status surface below), Decisions + hash-chained audit events for refusals (principle 6 — an unattended refusal at the boundary leaves the same explainable trail as anywhere else), and the two ADR-0019 §3 artifact-store credential entries (source-registry read; the §13.2 delivery-target write for the CDS drop) — plus the imported-promotion slice that IS the validation's reference data: the sync-journal entries and imported change rows carrying their M17.4(a)-verified `sourceRef.promotionManifest` (what `crossBoundaryManifestOf` reads for both the auto-relay trigger and the tarball authorized-set cross-check), the `imported_approval_evidence` rows, and pg-boss job state for the tick loop. No executor, deploy, or graph tables are ever written at a retrans.

**Validation is owned in-process — no runner class (owner-confirmed 2026-07-23).** The outposts' M17.4(a)+(b) verifies and the retrans's validate pass run inside their own server process, invoking the vendored cosign directly as a child process. The runner isolation pattern (`scp-runner-iac`, `scp-runner-scan`) exists for executing side-effectful tooling — tofu touching cloud APIs, Trivy loading a vulnerability DB; signature validation is a pure, deterministic check with nothing untrusted executing, so it lives in-process, exactly as the Ed25519 bundle verification always has.

Everything else in §13.1 — the inbox loop, the tick/poke trigger, the auto-relay — is behavior *of this slim profile*, not features carried over from the outpost.

**Invariants, restated so no reader blurs them:**

- **Metadata bundles stay byte-free** ([ADR-0009](../adr/0009-optional-poke-mode-federation.md)). The staging node moves **two separate channel artifacts** — the `.scpbundle` and the relay tarball — and never merges them. The bundle format does not change.
- **Retrans never terminates a promotion** ([ADR-0004](../adr/0004-service-naming-commander-outpost-retrans.md), unamended). Promotion state advances only at the commander and the receiving outpost. The staging node imports, relays, and drops; it decides nothing about the promotion.
- **The receiver's M17.4(a)+(b) run unweakened** ([ADR-0019](../adr/0019-artifact-byte-channel.md) §2 step 7). The staging node adds **no verification authority** — it is granted zero trust, exactly like the manual relay it automates. Nothing downstream changes because a loop carried the bytes instead of a human.
- **Inbox contents are untrusted.** File names and contents are data, not commands. The `resolveUnderDir` traversal guard on relay-tarball intake (PR #112 — file names inside the intake dir only, no traversal) survives automation, and every failing verify refuses with a block Decision, same as today. Automation changes who names the file, never what is trusted.
- **Every automated action writes the same Decisions and hash-chained audit events the manual CLI writes** (charter principle 6). An unattended import that refuses a tampered bundle produces the identical block Decision + audit event an operator-invoked one does.

**Status surface — and validate-gated confirmation (owner decision 2026-07-23, D4).** `bundle_transfers` rows (created → submitted → confirmed; *"purely observational bookkeeping — never consulted for authority/idempotency decisions"*, `apps/server/src/federation/bundle-transfers-repo.ts:7-13`) become the staging node's progress ledger: each automated hop updates the corresponding transfer row, so the commander UI/CLI — and the M16.1 boundary pipeline stages — show real per-hop state for an air-gapped peer's handoffs. The staging node **auto-confirms upward — but validate-gated, never blind**: the retrans validates the signatures FIRST (the verify paths above); only on success does it confirm the transfer upward (`bundle_transfers` confirm) and make the onward drop; on failure it refuses with a block Decision + hash-chained audit event and sends **no confirmation** — the transfer visibly stalls at the boundary rather than the ledger asserting a crossing that validation refused. Confirmation itself stays purely observational bookkeeping (never authority, per the repo header above); the gating governs when the ledger is allowed to say "crossed."

## 13.2 DeliveryTarget — filling exactly the ADR-0019 deferral, and no more

[ADR-0019](../adr/0019-artifact-byte-channel.md) explicitly did **not** design CDS handoff mechanics: *"drop-directory vs. diode transfer varies per CDS product; the relay's contract ends at 'signed tarball out / signed tarball in'"* (ADR-0019, Consequences). This section fills exactly that hole — how a signed file gets from SCP to the CDS's intake, and from the CDS's output to SCP — and nothing more. Everything past the drop (diode transfer, content inspection, the CDS product's review queue) is the org's CDS, out of scope (charter principle 1). **SCP hands bundles TO the CDS; it never operates the CDS.**

**Config shape — per-peer, with today's env as the instance-level fallback.** A `deliveryTarget` jsonb column beside `syncScope` on `federation_peers` (`apps/server/src/db/schema.ts:1022-1037` — the row today carries `role`, `baseUrl`, and the one `syncScope` jsonb), exposed through a zod-validated view following the M15.6 shape (`buildRegionalExecutorView` + fail-closed gate, `apps/server/src/coordination/regional-executors.ts` — validated config view, per-gap `problems`, never a silent misdeploy). Absent per-peer config, behavior is **exactly today's**: the instance-level env (`SCP_RELAY_OUT_DIR` / `SCP_RELAY_IN_DIR`, PR #112 `RelayConfig`) is the fallback. No migration of existing setups required.

**Per-peer dirs are operator-root-bounded (`SCP_DELIVERY_ROOTS`).** A per-peer `deliveryTarget` directory is a *data-supplied filesystem endpoint* set by an org admin with `federation:write`; absolute + traversal-free is not enough on a multi-tenant instance (any server-writable path — another org's `SCP_RELAY_IN_DIR`, any server-user-writable location — would otherwise be a legal drop target for the server-user `mkdir -p` + `writeFile`). So a per-peer dir is honored **only** at or under an operator-declared root in `SCP_DELIVERY_ROOTS` (comma/colon-separated absolutes; segment-safe prefix check), enforced in both places — refused at pair time (never stored) and re-checked fail-closed at resolution (a stored out-of-root dir is a named per-gap problem, never a silent env fallback). **UNSET = fail-closed** for any per-peer dir; the `SCP_RELAY_OUT_DIR`/`SCP_RELAY_IN_DIR` env fallback (operator-owned) is exempt, so single-org deploys need no new config. This is the same operator-allowlist pattern as `SCP_ARTIFACT_OCI_REGISTRY_HOSTS` (#108→#110, ADR-0019 §4). **13.2b flag:** the `s3-compatible` provider's `endpoint`/`bucket`/`prefix` are the same shape of data-supplied endpoint and MUST get the SAME operator-allowlist treatment (an operator-declared endpoint/bucket allowlist), fail-closed by default — the ADR-0019 §4 symmetry; do not ship the s3 endpoint without it.

**Providers:**

| Provider | What it is | Status |
|---|---|---|
| `filesystem` (default) | A directory path per direction — literally today's `SCP_RELAY_OUT_DIR`/`SCP_RELAY_IN_DIR` behavior, made per-peer | The write/read code exists (PR #112); only the per-peer resolution is new |
| `s3-compatible` | `endpoint`/`bucket`/`prefix` per direction — **operator config only, never bundle-steered** (the ADR-0019 §4 pattern: bundle data never picks an egress target; a peer configured to *require* a delivery target with none set **fails closed**) | Net-new runtime code — see honesty note |

**Honesty note — the Helm `objectStorage.s3` block is inert vocabulary, not behavior.** `deploy/helm/values.yaml:112-124` defines `objectStorage.provider: filesystem|s3` with an `s3.{bucket,region,endpoint,existingSecret}` block, and DESIGN.md §16 says "S3-compatible provider configurable" — but **no server code reads any storage config today**: when `provider != filesystem` the deployment mounts an `emptyDir` and injects no S3 env vars. This proposal **reuses the naming and config shape** so the vocabulary stays coherent; the runtime provider layer — an interface with filesystem and S3 put/list/get implementations — is **net-new code**, and we say so rather than citing the values block as existing behavior.

**The two seams, precisely:**

- **Write seam:** the single tarball-emission point — `buildRelayTarball` writes `SCP_RELAY_OUT_DIR/scp-relay-<id>.tar.gz` (PR #112) — plus the CLI's `--out` writes for `.scpbundle` files. A provider `put()` slots at exactly these points.
- **Read seam:** the §13.1 inbox loop lists a filesystem directory or an S3 prefix and feeds each object to the **existing** import paths. Verification semantics are unchanged; the one extraction refactor is noted in §13.1.

**Credentials.** New vault keys `delivery/<peer>/<target>` under the [ADR-0019](../adr/0019-artifact-byte-channel.md) §3 **artifact-store credential class** — an S3 bucket is a passive shelf, squarely the class §3 defined. **Write-scoped** for outbound drops, **read-scoped** for the inbox; same AES-256-GCM `secrets` vault, resolved at use, injected via mode-0600 scratch config, never argv/logs/Decisions — the exact discipline of PR #112's `relay/source-read/<host>` / `relay/dest-push/<host>` keys. ADR-0019 §3 recorded the credential-class expansion honestly; this adds *targets within* that class, not a new class.

**Air-gap and principle 4.** The S3 client is **vendored at build time** (no runtime fetch, principle 5), and S3 remains **optional**: filesystem is the default, the two-container floor is unchanged, and **PostgreSQL stays the only required stateful dependency** (principle 4) — a delivery bucket is the org's/CDS's infrastructure, not SCP's.

**Client (owner decision 2026-07-23, D3): `@aws-sdk/client-s3` (AWS SDK v3) IS the default.** The honest rationale: relay tarballs are multi-GB, and the SDK's managed **multipart upload** (`@aws-sdk/lib-storage`) is the difference between a working drop and a hand-rolled one; first-party **SigV4 correctness** (chunked/streaming signing edge cases included) and built-in retry/backoff are exactly the surface an unattended boundary loop must not get subtly wrong. The vendoring cost is real, but the air-gap principle constrains **runtime network calls, not dependency size** — the SDK is vendored at build time like everything else (principle 5). S3-compatibles (MinIO etc.) are covered via `endpoint` override + `forcePathStyle`. (The draft recommended `aws4fetch` for its audit surface; the owner chose the SDK — see the Decisions record.)

## 13.3 Managed scanning — `scp-managed-scan`, a first-class service of the commander's promotion process

**Reframed by owner decision (2026-07-23, in writing): managed scanning is not a Mode-C fallback for pipeline-less orgs — it is how the commander promotes.** In the owner's words: *"the commander is what executes the scans and signatures as part of the promotion process. It would use Trivy, OpenSCAP, and other additional scanning methods provided and assigned an artifact type."* The promotion process at the commander is:

```
scan (scanners selected per artifact type — scanner registry, below)
  ──▶ evaluate (vs the M17.5 scan-requirement resolution)
  ──▶ sign (the promotion manifest, M17.3 E6 — only if scans pass)
  ──▶ export
```

E6's evidence therefore exists **by construction** for every promotion — an org with no pipeline scanner promotes exactly like one with a full CI fleet. **Org-pipeline scan evidence remains a supported alternate ingress**, through the existing shapes — the `scan-result-control` pull path and the report paths — and the gates consume evidence identically from either source: where valid org-pipeline evidence already covers an artifact's digest, the promotion scan step consumes it instead of re-producing it. The signing model is untouched: the **executor** still cosign-signs artifacts + SBOM at build and the **commander** still cosign-signs **only its own promotion manifest** ([ADR-0015 §5](../adr/0015-cosign-cross-boundary-signing.md)) — "executes the signatures as part of the promotion process" is the E6 manifest signing that already ships; what is new is that the commander now also **runs the scan step** that authorizes it.

The runner pattern is `scp-managed-iac`'s (DESIGN.md §12), running at the commander:

- **A thin orchestrator plugin** (`scp-managed-scan`) behind the standard executor interface. It schedules runs, collects results, and publishes evidence; it contains no scanner.
- **A separate `scp-runner-scan` image** carrying digest-pinned **Trivy** and **OpenSCAP (`oscap`)** — the tools exist **only** in the runner image, exactly as `tofu` exists only in `scp-runner-iac` (*"the only place `tofu` exists in the whole system"*, `apps/runner-iac/README.md`). Pins follow the `tools/*/pin.env` vendoring discipline with drift-failing tests (the #111 skopeo pattern): `tools/trivy/pin.env`, `tools/openscap/pin.env`.
- **Ephemeral, single-shot containers** — a Kubernetes Job in production, `docker run` under compose — with `--network none` **except** an operator-allowlisted registry pull for the subject artifact's bytes (the ADR-0019 §4 allowlist treatment: the registry host must be operator-allowlisted; nothing in tenant data steers egress). No phone-home, ever (principle 5): vulnerability data is pre-loaded (below), and the scanner runs with DB downloads disabled.

**The scanner registry — methods assigned to artifact types (graph-native).** Scanning methods are **registry data, assigned to artifact types** — registry rows, not new top-level tables (charter principle 2: new concepts arrive as relationship/policy/registry data). The promotion scan step reads each artifact's type and selects the assigned scanner(s); an artifact type with **no** assigned scanner and **no** org-supplied evidence refuses at E6 exactly as today — fail-closed, unchanged. The design rows:

| Scanner | Assigned artifact types (registry rows) |
|---|---|
| `trivy` | container images; **machine images** (VM disk images / AMIs); filesystems; OS packages |
| `openscap` | OS images, evaluated against assigned SCAP compliance profiles |
| *(future scanner plugins)* | whatever types their registry rows assign — the registry is the extension point |

**M13 scope (owner decision 2026-07-23, D2): image-only — where "image" INCLUDES machine images.** M13 ships the image-type assignments: **container images** and **machine images** (AMIs, VM disk images). Machine images are scanned with Trivy's **`trivy vm`** — an **experimental** Trivy feature, and the registry row carries its actual support bounds honestly rather than promising every disk format: in **disk-file form** for air-gap (local **VMDK**, `streamOptimized` subformat; OVA/VHD/VHDX/QCOW2 are not currently supported upstream), and in **AWS API form** when connected (`ami:<id>` / `ebs:<snapshot-id>` via the EBS direct APIs; marketplace AMIs excluded); filesystem coverage ext2/3/4 + XFS over MBR/GPT partitions (no LVM/ZFS). The filesystem/package assignments for `trivy` are registry-design headroom, not M13 scope. **No live-host scanning in M13**: host-level SCAP against running infrastructure is `scp-runner-ops`-adjacent, host-reaching territory (a different charter conversation) and stays out — `openscap` in M13 evaluates **OS images** against compliance profiles, never live hosts.

**Evidence lives at the commander — ONLY (owner decision 2026-07-23, D5) — and exits through the existing shapes, zero gate changes.** `ScanEvidenceSchema`'s `scanner` field is `z.literal("trivy")` *"a field, not a constant, so the evidence is self-describing in a Decision and a future second scanner slots in without a shape change"* (`packages/schemas/src/supply-chain.ts:180-181`) — it was designed for exactly this: the literal grows to an enum (`"trivy" | "openscap"`). The M17.5 six-tier threshold resolution (`apps/server/src/governance/scan-requirements.ts`) and the E6 export gate are **untouched** — they consume evidence identically whether an org's own pipeline step or the managed runner produced it. Where the evidence *bytes* live is decided:

- **Managed-scan evidence is commander-resident.** The runner's result JSON (and OpenSCAP XCCDF results) land in the **commander's evidence store**, and `ScanEvidenceSchema` verdicts are parsed from it there — Postgres-backed, so principle 4 is intact (no new stateful service; an evidence table is a registry-shaped addition, not a blob service). **There is NO Gitea prerequisite anywhere for evidence** — not at the source, not at the boundary, not at the receiver. (The draft's Gitea-hosted placement is withdrawn; OQ5 is DECIDED — see the Decisions record.)
- **Outposts and retrans NEVER store or read scan evidence.** They validate the commander's signature, which is the **transitive proof of scan-pass** — in the owner's words, an artifact *"would only have a valid one signed by the commander if they passed scans."* The M17.4(a)+(b) verify hops already need exactly nothing beyond the manifest + signatures they verify today; evidence travels nowhere.
- **The org-pipeline ingress is unchanged.** `scan-result-control` still pulls org-produced evidence from the per-binding operator-configured URL (`ctx.http`); that evidence lives wherever the org's pipeline put it, and what SCP persists of it stays reference + parsed verdict, as today. Commander-residency governs **managed**-scan output.

One honest note: the draft leaned on *"SCP has no blob storage anywhere … reference in, reference out"* (`supply-chain.ts:224-227`) to justify pushing evidence bytes out to Gitea; the owner's decision **evolves that posture for managed-scan evidence specifically** — the commander holds the evidence it itself produces (it is the evidence's origin, not a cache of someone else's bytes). The header-comment revision is 13.3a build work, alongside the `supply-chain.ts:18` revision noted under Charter ground below.

**Scan-once — AT THE COMMANDER, before signing (owner decision 2026-07-23; supersedes scan-at-source for the signing flow).** The 2026-07-22 draft ran the managed scanner *"where artifacts are built"* — the originating outpost, under [ADR-0017](../adr/0017-ownership-refinement.md) §2 build devolution. **That placement is superseded**: the scan is part of the promotion process, and the promotion process is the commander's — so the scan executes **at the commander**, which pulls the artifact bytes **by digest** over allowlisted channels that already exist for exactly this (the [ADR-0019](../adr/0019-artifact-byte-channel.md) byte channel; the ADR-0019 §4 / `SCP_ARTIFACT_OCI_REGISTRY_HOSTS` registry-host allowlist — nothing in tenant data steers where bytes are pulled from). **"Scan-once" now means once-at-the-commander-before-signing**, and the signature is the transitive proof that carries the scan-pass downstream. What does NOT change: **downstream never re-scans.** Receivers — retrans and outposts alike — validate signatures ([ADR-0011](../adr/0011-universal-outpost-validation.md)'s universal validation; [ADR-0010](../adr/0010-outpost-local-artifact-infra.md)'s receivers-stay-light), exactly as before. The *location* of the one scan moved; the *number* of scans (one) and the no-re-scan rule did not.

**Why once is the right number (owner rationale, 2026-07-23): scan results are time-decaying.** New CVEs are published continuously, so an artifact's scan results inevitably worsen over time with no change to the artifact. If every promotion hop re-scanned, outposts farther down the promotion line would systematically face worse results than hops nearer the commander — the deep (air-gapped) end of the chain would be penalized for being far, and a long chain might never promote at all. Scanning and signing **once at initial promotion** pins the attestation to the moment that matters: *at the time of initial promotion, the artifact passed the then-current requirements* — and signature validation, unlike scan results, does not decay. The rule is per **promotion journey**, not per artifact lifetime: promoting the same artifact again later is a new initial promotion and gets a fresh scan against current requirements — fairness across the chain without grandfathering stale artifacts into new journeys.

The apparent tension the draft resolved — *if M13 completes the air-gap story, shouldn't the CDS staging node scan what crosses?* — resolves the same way, one step simpler: **no.** Every artifact that legitimately reaches a staging node was scanned at the commander (or covered by valid org-pipeline evidence) before signing — E6 hard-refuses any export lacking passing digest-bound evidence — so a staging-node scan would be a re-scan of already-signed-for evidence. The boundary node stays exactly as thin as ADR-0004 declared it, and the §13.1 deployment profile makes that structural: the retrans profile ships cosign and skopeo only; the `scp-runner-scan` image never lands on a staging node.

**ADR reconciliation — honest, not rewritten.** Three ADR statements are evolved by this owner decision (2026-07-23), and we mark the evolution rather than silently editing history: [ADR-0013](../adr/0013-supply-chain-scan-sbom-manifest.md)'s scan-at-source (the coordinated Trivy step at build) becomes the org-pipeline **alternate** ingress rather than the only model; [ADR-0010](../adr/0010-outpost-local-artifact-infra.md)'s "scan-at-source … at the origin outpost" reading likewise (its receiver-side half — verify the signature, never re-scan — is untouched); and [ADR-0017](../adr/0017-ownership-refinement.md) §2's "the commander … does **not** run the scan" is superseded — the commander's cross-boundary gate now **includes the promotion scan step** (the gate-only role widens by exactly this step; build stays devolved, and the commander still never runs build). The ADR files themselves are not rewritten in this PR; per the working convention (significant decisions get an ADR after approval), the follow-up ADR recording first-class commander scanning has landed: **[ADR-0020](../adr/0020-first-class-commander-scanning.md)** (Accepted 2026-07-23), which enumerates exactly what it evolves vs preserves in 0010/0013/0017.

**Default-permissive means ADOPTION, never gate-weakening.** Resolving the second tension explicitly, restated under the first-class framing: "default-permissive" describes when a scan is *scheduled*, not what the gates *enforce*. A domain-local change that never crosses a boundary triggers no promotion and therefore no promotion scan — an org that never crosses pays nothing, matching ADR-0013's "boundary-authorization gate, not a general quality gate" framing and ADR-0018's path-based exemption. A **cross-boundary promotion always gets the scan step** — that is what first-class means — with valid org-pipeline evidence consumed in place of a managed run where it already covers the digest. The M17.5 fail-closed floor rules and E6 apply **unchanged**. The promotion scan step adds a way to *satisfy* the gates; it adds no way to loosen them.

**Offline scanner data — crosses boundaries as `blob`, not as a scanned subject.** Upstream, the Trivy vulnerability DB is an OCI artifact at `ghcr.io`, and connected instances may still refresh from it directly via the operator-invoked, allowlisted registry pull (the vendored-skopeo machinery, #111). But E6 (`evaluatePromotionScanGate`, `promotion-repo.ts:104-140`) hard-refuses **every** non-`blob` artifact crossing a boundary unless it carries its **own** passing, digest-bound scan evidence — and a DB cannot meaningfully scan itself, so typing the DB as an ordinary OCI artifact on the cross-boundary channel would make it E6-unsatisfiable by construction, not merely awkward. So when scanner data crosses a boundary, both the trivy-db **and** the SCAP data-stream content cross as **`type: "blob"` artifacts**: cosign-signed, digest-bound, carried by the relay's existing blob path (`retrans-relay.ts` blob handling, PR #112, `feat/m15-5c3-retrans-relay`) or the air-gap bundle's blob path. This is E6-exempt for the same reason the SBOM is — scan-adjacent data, not a scanned deployable subject (the SBOM is the scan's *output*, `promotion-repo.ts:106`; the DB is its *input*) — the exemption spelled out where `evaluatePromotionScanGate` scopes itself to `substantiveArtifacts` (everything except `type: "blob"`, `promotion-repo.ts:104-116`, filter at `promotion-repo.ts:253`). The refresh flow extracts/repackages the upstream OCI-form DB into that blob shape for the channel; the OCI form itself never needs to cross a boundary. E6 stays completely unmodified — it never evaluates the DB as a substantive artifact, so the export passes with **zero gate-code changes**; the DB's "pass" at the boundary is a vacuous exemption, not scan evidence, and the refresh runbook says so explicitly. Air-gapped instances refresh via the promotion channel's blob path at whatever cadence the org promotes — the destination that matters is the **air-gapped commander's** runner (only commanders scan, D1/D5; outposts and retrans never consume scanner data). No new transport, no new trust: the blob is verified digest-bound at the receiving end like every other blob artifact. (This is genuinely new *cadence* ground — the `pin.env` discipline refreshes by commit, and no periodically-refreshable data artifact exists in the system today; the refresh runbook is increment 13.3b.)

**Increment order: Trivy first, OpenSCAP second.** Trivy result JSON already maps onto `ScanEvidenceSchema` (severity counts, digest binding), and Trivy covers both image sub-kinds in the D2 scope (container images; machine images via `trivy vm`, support bounds above). OpenSCAP's XCCDF rule results need an explicit mapping (failed-rule severities → the four severity counts the threshold model acts on) — designed and landed as the second half of 13.3a, not hand-waved into the first.

**CHARTER GROUND — the extension is APPROVED (owner, 2026-07-23, in writing).** The charter's **Managed Execution Exception** enumerates managed-IaC (2026-07-08) plus three host-reaching classes (2026-07-12), and states: *"Extending this class allowlist requires owner sign-off."* That sign-off has been given: `scp-managed-scan` is enumerated as a second non-host-reaching managed class, and **this PR amends PROJECT_CHARTER.md accordingly** (amendment dated 2026-07-23, following the exception section's existing form: standard executor interface; isolated single-shot ephemeral runners from a separate `scp-runner-scan` image; scoped vaulted credentials; no host reach; `--network none` except operator-allowlisted registry pulls). The reconciliations that come with it, said plainly:

- The [ADR-0017](../adr/0017-ownership-refinement.md) tightening — the commander "consumes the coordinated Trivy step's verdict as gate evidence… It does **not** run the scan" (ADR-0017 §2, lines 73–75) — and `supply-chain.ts:18`'s *"SCP NEVER runs Trivy"* are **preserved for the GATE**: the governance engines still only consume evidence; E6 never runs a scanner. But the **runner genuinely executes scans** in SCP-shipped containers — which is precisely why this took a charter amendment, not a reinterpretation. The header comment is revised in 13.3a to "the SCP *gate* never runs Trivy; the charter-enumerated `scp-managed-scan` runner does, as the promotion scan step." The 13.3a comment sweep is a **census, not an instance fix** — alongside `supply-chain.ts:18` and `:224-227` it covers the other stale "trust scan-at-source" comments: `apps/server/src/federation/artifact-verify.ts:26`, `apps/server/src/federation/promotion-repo.ts:392`, and `apps/server/src/federation/federation.integration.test.ts:1419` (all three are receiver-side, so their never-re-scan meaning survives; only the scan-*location* wording is stale).
- The charter's *"Managed execution is never a default; the six-gate boundary test is the only route into it"* governs managed execution **of changes**. The managed scanner **executes no change** — it is read-only with respect to the scanned subject, and the amendment says so in terms — so running it as a default step of promotion does not make change-execution a default. The class enters the allowlist by explicit owner enumeration, the route the charter reserves to the owner.
- **The draft's BYO-only alternative was considered and declined.** Shipping only a documented coordinated scan-step recipe (no charter change) would have left pipeline-less orgs unable to promote across boundaries until they adopted a build engine — exactly the population the Managed Execution Exception exists to serve. The owner instead made managed scanning first-class; org-pipeline evidence stays supported as the alternate ingress.

## Decisions (owner, 2026-07-23 — all given in writing)

The five open questions of the 2026-07-22 draft are all **DECIDED**. Recorded here with what changed against the draft's recommendations:

1. **Charter extension + first-class managed scanning — APPROVED, and widened (D1).** The Managed Execution Exception is extended to enumerate `scp-managed-scan`; **this PR amends PROJECT_CHARTER.md** (amendment dated 2026-07-23, following the exception section's existing enumerated-class form). The owner went further than the draft asked: managed scanning is **not** a Mode-C fallback for pipeline-less orgs but a **first-class commander service** — *"the commander is what executes the scans and signatures as part of the promotion process. It would use Trivy, OpenSCAP, and other additional scanning methods provided and assigned an artifact type"* — with a **scanner registry** assigning scanning methods to artifact types (§13.3). Org-pipeline scan evidence remains a supported alternate ingress. Decided with it, superseding the draft's scan-at-the-origin-outpost placement: the scan executes **at the commander**, which pulls artifact bytes by digest over allowlisted channels — **scan-once = once-at-the-commander-before-signing**; the signature is the transitive proof; downstream never re-scans. The ADR-0010/0013/0017 evolutions are marked in §13.3; the follow-up ADR is **[ADR-0020](../adr/0020-first-class-commander-scanning.md)** (landed at M13 approval, Accepted 2026-07-23).
2. **OpenSCAP / scanning scope — image-only for M13, where "image" INCLUDES machine images (D2).** AMIs and VM disk images are in scope as images, scanned via Trivy's experimental `trivy vm` (disk-file form — local streamOptimized VMDK — for air-gap; `ami:`/`ebs:` via the EBS direct APIs when connected; support bounds recorded honestly in §13.3). **No live-host scanning in M13** — host-level SCAP against running infrastructure is `scp-runner-ops`-adjacent and out. (Matches the draft recommendation, with the machine-image widening owner-added.)
3. **S3 client — `@aws-sdk/client-s3` (AWS SDK v3) IS the default (D3).** Managed multipart upload for multi-GB relay tarballs, first-party SigV4 correctness, built-in retries; vendored at build time — the air-gap principle constrains runtime network, not dependency size; `endpoint` override + `forcePathStyle` for S3-compatibles (MinIO etc.) (§13.2). **The draft's `aws4fetch` recommendation is replaced.**
4. **Transfer confirmation — auto-confirm, VALIDATE-GATED, never blind (D4).** The retrans validates the signature FIRST; on success it confirms the transfer upward (`bundle_transfers` confirm) and makes the onward drop; on failure → a refusal block Decision + hash-chained audit event and **no confirmation** (§13.1 status surface). Confirmation stays observational bookkeeping, never authority — the gating governs when the ledger may say "crossed."
5. **Evidence bytes — commander-resident, ONLY (D5).** Managed-scan evidence lives in the **commander's evidence store**; **no Gitea prerequisite anywhere for evidence** — the draft's Gitea-by-default placement is **withdrawn**. Outposts and retrans never store or read scan evidence — they validate the commander's signature, the transitive proof of scan-pass: *"it would only have a valid one signed by the commander if they passed scans"* (§13.3).

Nothing else in this proposal is owner-blocking; no question remains open at the owner level. The proposal as a whole (build plan + phasing) remains pending ordinary review.
