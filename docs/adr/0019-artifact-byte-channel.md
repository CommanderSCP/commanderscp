# ADR-0019: The artifact byte channel — per-tier transport, the retrans validate-then-relay, and the artifact-store credential class

**Status:** Accepted (owner mandate 2026-07-20 — finish **all** of M15/M17: the automated retrans relay is **required**, no longer "if operator-loaded bytes prove insufficient")
**Context doc:** [docs/proposals/promotion-and-execution-model.md §5](../proposals/promotion-and-execution-model.md) (metadata vs. bytes)
**Relates to:** [ADR-0004](0004-service-naming-commander-outpost-retrans.md) (the declared `retrans` validate-and-relay role this ADR builds out); [ADR-0009](0009-optional-poke-mode-federation.md) (metadata-only bundles — the invariant this ADR does **not** touch; poke propagates via the retrans chain); [ADR-0010](0010-outpost-local-artifact-infra.md) (the outpost's local Gitea the bytes land in); [ADR-0011](0011-universal-outpost-validation.md) (the receiving outpost always validates — the relay does not replace it); [ADR-0012](0012-registry-consolidation.md) (Gitea is the registry at both ends; the commander never pushes — the deploy tool pulls); [ADR-0015](0015-cosign-cross-boundary-signing.md) §6 (the split verify hop — per-artifact verify happens where the bytes land; this ADR resolves the "still-unresolved artifact-bytes transport channel"); M17.4(a) #106 + M17.4(b) #108 (`apps/server/src/federation/artifact-verify.ts` — the digest-bound, fail-closed verify machinery the relay's VALIDATE step reuses); DESIGN.md §12 (`scp-managed-iac` — the sanctioned vaulted-scoped-credential pattern the new credential class is modeled on); charter principle 1 (coordinate, not execute)

## Context

Federation/promotion bundles are **metadata-only** ([ADR-0009](0009-optional-poke-mode-federation.md); `packages/schemas/src/federation.ts`) — digests, signatures, SBOM refs, the cosign-signed promotion manifest — never artifact bytes. Artifact **bytes** travel on a **separate channel** ([promotion-and-execution-model.md §5](../proposals/promotion-and-execution-model.md)). Until now that channel was only partly decided: the owner chose **operator-loaded-first** (outpost-local-artifact-infra.md, decision 2, 2026-07-18), and the retrans/CDS byte-relay sat behind "M15.5 (optional/later) — if operator-loaded bytes prove insufficient." The owner has now mandated finishing **all** of M15 — the automated relay is **required**, and this ADR is its design.

What changed underneath the old deferral is that **the hard part already shipped**:

- **M17.4 is complete.** (a) manifest-signature + arrived-set-equality verify at bundle import (#106, `promotion-repo.ts::importPromotionBundle`), and (b) per-artifact, **digest-bound**, fail-closed byte verify as a pre-deploy gate (#108, `coordination/pre-deploy-gate.ts` + `federation/artifact-verify.ts::verifyAuthorizedArtifactSet`). The relay's VALIDATE step is **this same machinery**, not new crypto.
- **The transport mechanics exist at ~80% in `deploy/airgap`.** `skopeo.ts::copyToOciLayout` supports `sourceType: "docker"` — i.e. pulling **from a registry** into an OCI layout, exactly the relay's ingest direction. `build-bundle.ts` produces an OCI-layout tarball with a per-digest manifest, `CHECKSUMS.txt`, and a cosign signature. The `.scpbundle` pattern already moves signed files across air gaps. `install.sh` already pushes images **by digest** into a destination registry and **re-inspects** the pushed digest to confirm byte-for-byte identity. The relay recomposes these proven pieces; it does not invent a transport.
- **Retrans is a declared, unbuilt role.** [ADR-0004](0004-service-naming-commander-outpost-retrans.md) defined `retrans` as validate-then-relay at a CDS boundary — never originates config, never holds local authoritative objects, **never terminates a promotion** — but shipped it naming-only (no relay logic; the M15.4 role matrix bundles it **zero** backends). This ADR gives the role its behavior.
- **#108 left an acknowledged residual.** Blob byte fetches are gated by `SCP_ARTIFACT_BLOB_BASE_URLS` (operator allowlist; **unset = every blob URL rejected**, fail-closed), but `cosign verify` for OCI artifacts still dials the **bundle-supplied OCI registry host**. Digest-binding means no *substitution* is possible, but the egress-steering asymmetry stands: a symmetric OCI-registry-**host** allowlist was the flagged follow-up. The byte-channel config surface defined here (§4) is its natural home.

One question this raises is charter-shaped and load-bearing: a relay that pulls and pushes artifact bytes must **hold credentials** — and SCP's principle 1 says the platform does not hold credentials to the infrastructure execution systems manage. §3 resolves this by naming what these credentials actually are.

## Decision

### 1. Per-tier byte transport — one table, three answers

The byte channel is **per trust tier**. SCP's own code footprint differs sharply by tier — and for the commercial tier it is deliberately **zero transport code**:

| Tier | Transport | SCP's role | Verify | Status |
|---|---|---|---|---|
| **Commercial / connected** | The connected outpost's **deploy tool pulls by digest** from the **commander-hosted registry** (Gitea, [ADR-0012](0012-registry-consolidation.md)); the commander **never pushes** | **Hosting + verify only — SCP writes NO transport code.** The registry serves; the deploy tool (Argo CD et al.) does what it already does | M17.4(a) at import; M17.4(b) pre-deploy byte verify (DONE, #108) | **Documented by this ADR — nothing to build** |
| **High / air-gap — MVP** | **Operator-loaded media** (the existing air-gap bundle path: skopeo OCI layout → walked across → pushed into the outpost's **local Gitea**, [ADR-0010](0010-outpost-local-artifact-infra.md)) | Verify what arrived; coordinate the local Gitea | M17.4(a)+(b) — the pre-deploy gate verifies operator-loaded bytes digest-bound, fail-closed | **DONE** (M17.4b shipped; this row is the operator-loaded path it was built for) |
| **High / air-gap — automated** | The **retrans validate-then-relay** (§2) — pull from the source registry, validate, package a signed OCI-layout tarball, carry it across the CDS as a file, push into the destination local Gitea by digest | Build the relay (the one net-new piece) | Retrans validates **before** crossing; the receiving outpost **still** runs M17.4(a)+(b) — defense in depth, §2 | **To build — M15.5(c)** |

Pure **sneakernet** (no CDS data path at all) remains the MVP row forever — operator-loaded media *is* its automated ceiling.

### 2. The retrans validate-then-relay — pipeline of proven pieces

The relay is the [ADR-0004](0004-service-naming-commander-outpost-retrans.md) role made real: **validate-then-relay, nothing more**. Its pipeline, step by step, with the existing machinery each step reuses:

1. **Resolve** the authorized artifact set — the digests enumerated by the commander's cosign-signed **promotion manifest** ([ADR-0015](0015-cosign-cross-boundary-signing.md) §2; typed `artifacts[]`). The manifest is the *only* source of what may cross; the relay never invents or accepts an artifact outside it.
2. **Pull** each artifact's bytes from the **source registry** (`skopeo` — the `copyToOciLayout` path with `sourceType: "docker"`, i.e. registry → OCI layout; blobs via their `location` under the §4 allowlist).
3. **Validate** — run the M17.4 machinery (`verifyAuthorizedArtifactSet`: manifest-signature verify, arrived-set equality, per-artifact digest-bound cosign verify against the exporter's distributed pubkey, keyful/offline). **Fail-closed: a failing, tampered, unauthorized, or missing artifact never crosses the CDS.** A failure persists a block Decision + hash-chained audit event, like every gate.
4. **Package** a signed **OCI-layout tarball** (the `build-bundle.ts` machinery: OCI layout + per-digest manifest + `CHECKSUMS.txt` + cosign signature).
5. **Relay** the tarball across the CDS **as a file** — the `.scpbundle` transport pattern, unchanged in kind: signed file out, signed file in. (Poke-mode's retrans-chain hops, [ADR-0009](0009-optional-poke-mode-federation.md), are the notification counterpart of this same data path.)
6. **Push** each image into the destination outpost's **local Gitea by digest and re-inspect** the landed digest (the `install.sh` pattern — a registry push cannot silently alter what was verified).
7. **Receiver re-verifies.** The receiving outpost **still runs M17.4(a) at import and M17.4(b) pre-deploy**, exactly as if an operator had loaded the bytes. **The relay is granted zero trust:** it is an optimization of the sneakernet leg, not a verification authority. Nothing downstream is weakened because a relay carried the bytes.

Two invariants restated so no future reader blurs them:

- **The relay does not put bytes in bundles.** Federation/promotion bundles remain **metadata-only** ([ADR-0009](0009-optional-poke-mode-federation.md)) — the byte tarball is a **separate channel artifact**, produced and consumed by the relay, never a change to the bundle format.
- **The relay never terminates a promotion.** Promotion state advances only at the commander and the receiving outpost; the retrans holds no local authoritative objects (ADR-0004, unamended).

### 3. The new credential class — vaulted, scoped **artifact-store** credentials (the load-bearing decision)

**The retrans relay holds credentials: source-registry READ + destination-Gitea PUSH. These are ARTIFACT-STORE credentials — registry credentials — NOT credentials to the infrastructure execution systems manage.** This is the decision that makes the relay charter-compatible, so it is stated as prominently as the pipeline itself.

Charter principle 1 forbids SCP holding credentials to the infrastructure that execution systems manage — kube-creds, cloud-provider creds, the things a deploy tool wields. The relay touches **none** of that. Its credentials let it read bytes from one artifact store and write bytes into another — the same *category* of credential as the scoped Gitea API token SCP already holds to coordinate a bundled backend, extended from API-coordination scope to byte read/push scope. An artifact store is a passive shelf, not an execution system; pushing a verified image onto a shelf deploys nothing.

The class is modeled on the **`scp-managed-iac` pattern** (DESIGN.md §12; the charter's sanctioned Managed Execution Exception credential handling): vaulted, scoped, injected only for the duration of use. Concretely:

- **Scoping — per-peer AND per-registry.** A relay instance serving the `airgap-1` boundary holds (i) a **read-only** pull credential for the specific source registry (the commander's Gitea, or the originating outpost's under [ADR-0017](0017-ownership-refinement.md) build-devolution) and (ii) a **push-only** credential for the specific destination Gitea, scoped to the artifact repositories the promotion pipeline uses — not org-admin, not instance-admin, no delete. One boundary's credentials grant nothing at any other boundary.
- **Storage — the existing `secrets` table, exactly as executor credentials are held today.** Executor bindings already resolve credentials via `executor_bindings.secretRefs` → the AES-256-GCM `secrets` table (`secrets/crypto.ts`, `secrets/secrets-repo.ts`); the managed-IaC runner's infra credentials live there too. The relay's credentials use the **same vault, same envelope, same resolution seam** — a dedicated table is explicitly rejected (Alternatives) because a second credential store is a second thing to secure, audit, and rotate for zero added isolation.
- **Rotation.** Registry credentials are operator-rotated like any secret in the vault: write the new value, the next relay run resolves it — no relay restart, no downtime coupling. Both endpoints (source Gitea, destination Gitea) support token revocation independently; revoking either side fail-stops the relay (it can no longer pull or push), it does not fail-open.
- **Blast radius, stated honestly.** A fully compromised relay credential set can: read artifacts it could already read (they cross the CDS anyway) and push artifacts into the destination registry. The second is why step 7 exists: **pushed bytes deploy nothing** — the receiving outpost's M17.4(b) digest-bound gate blocks any pushed artifact that is not in a commander-signed manifest with a valid origin signature. The credential compromise ceiling is registry pollution, not deployment.

### 4. The byte-channel config surface — both allowlists live here

The relay and the pre-deploy verify share one operator-configured egress surface, and this ADR fixes both halves of it:

- **`SCP_ARTIFACT_BLOB_BASE_URLS`** (shipped in #108) — comma-separated, operator-configured base URLs that blob `location`/`signatureRef` fetches must fall under. **Unset = every blob URL rejected, fail-closed**; bundle-supplied URLs never steer egress. Unchanged; restated here because this config surface is now the byte channel's home.
- **`SCP_ARTIFACT_OCI_REGISTRY_HOSTS`** (**new — closes the #108 acknowledged residual**) — the **symmetric** comma-separated allowlist of OCI registry **hosts** that per-artifact `cosign verify` (and the relay's skopeo pull) may dial. Today the OCI verify dials the bundle-supplied registry host in the digest-pinned ref: digest-binding prevents artifact **substitution**, but a hostile bundle can still pick the egress **target**. With this variable, an OCI `location` whose host is not allowlisted is rejected before any dial — same shape, same guard site (`artifact-verify.ts`), same failure surface as the blob guard. **Unset = every OCI location rejected, fail-closed — deliberately symmetric with the blob allowlist**, and deliberately a behavior change for existing operator-loaded setups (they must set the variable on upgrade; a loud release note, not a silent default — Consequences).
- **`SCP_ARTIFACT_INSECURE_HOSTS`** (added post-#116, closing the gate/relay TLS asymmetry) — the pre-deploy verify's per-host TLS-off allowlist, the gate-side sibling of the relay's `SCP_RELAY_INSECURE_HOSTS` (a deliberately SEPARATE variable — different subsystem, same parse): comma-separated registry `host[:port]` entries the M17.4(b) `cosign verify` may dial with `--allow-insecure-registry`. Before this, the gate granted TLS-off blanket to every allowlisted host; now TLS-off is per-host operator opt-in, and unlisted hosts keep full TLS verification. **Unset = TLS verification everywhere** — the secure default, safe because the cosign signature (not transport TLS) is the trust anchor; a plain-HTTP outpost-local registry must be listed explicitly (release-noted, like the egress allowlist above).
- The relay's own **pull** side uses the same two allowlists (its egress is operator-configured, never bundle-steered), and its **push** side needs no allowlist — the destination is the relay's own configured Gitea, not bundle-supplied data.

## Charter alignment

- **Coordinate, not execute (principle 1):** the relay is **federation transport** — the [ADR-0004](0004-service-naming-commander-outpost-retrans.md)-declared retrans role doing exactly what was declared: validate, then relay. It is the same *category* of act as the metadata-bundle relay and the `.scpbundle` file walk SCP already performs; the payload is artifact bytes instead of journal rows. It never deploys, provisions, or runs tests; it never terminates a promotion; the deploy verbs stay observe/trigger/status/abort, untouched. Its credentials are **artifact-store** credentials (§3), not credentials to execution-managed infrastructure — the credential-asymmetry invariant holds: the deploy tools keep their kube/cloud creds, SCP holds none of them.
- **Graph-native (principle 2):** the source/destination registries are the existing `execution-system` graph objects ([ADR-0010](0010-outpost-local-artifact-infra.md)/M15.3); relay runs persist Decisions and audit events; no new top-level concept beyond behavior for the already-declared role.
- **PostgreSQL-only required dependency (principle 4):** the relay adds no stateful service — cosign is already vendored (digest-pinned, M17.3 E1) and **skopeo must be vendored into the runtime image as part of this build** (pinned, the same M17.3 pattern; today it is an operator-`PATH` prerequisite, BUILD_AND_TEST §1); binaries in the image, state in Postgres, the tarball a file.
- **Air-gap first-class (principle 5):** this ADR **is** that principle's byte half — fully offline: keyful/offline cosign, offline-distributed pubkey, file-based CDS crossing, no external fetch.
- **Explainability (principle 6):** every relay validate verdict persists a Decision; a refused artifact yields a block Decision + hash-chained audit event naming the digest and the failing check, at the retrans **and** again at the outpost if re-presented.

## Alternatives considered

- **Artifact bytes inside federation bundles (rejected — invariant).** Would collapse the metadata/byte split [ADR-0009](0009-optional-poke-mode-federation.md) rests on, bloat every bundle with the largest artifact's bytes, and force byte handling into every import path. The byte channel stays separate; the bundle format does not change.
- **Operator-loaded-only, forever (superseded by owner mandate).** The prior stance. Honest and shipped (M17.4b verifies it), but it caps air-gap promotion throughput at sneakernet cadence and burns operator time per promotion. The owner's 2026-07-20 finish-all-of-M15 mandate ends the deferral; operator-loaded remains the MVP row and the sneakernet fallback.
- **Commander pushes bytes to outposts (rejected).** Inverts the pull model ([ADR-0012](0012-registry-consolidation.md): the commander never pushes), requires commander→outpost data connectivity that air-gap tiers structurally lack, and hands the commander push credentials to every outpost registry — the maximal credential blast radius instead of the per-boundary minimal one (§3).
- **A "dumb pipe" relay — carry without validating (rejected).** Cheaper, but it forfeits the CDS-boundary property that **nothing invalid enters the air-gapped domain** ([ADR-0011](0011-universal-outpost-validation.md); promotion-and-execution-model.md §6 "full strength at every hop") and contradicts ADR-0004's validate-then-relay contract. The validate step is nearly free — the M17.4 machinery already exists.
- **A dedicated credential table for relay credentials (rejected).** A second store to secure, audit, and rotate, for zero isolation gain over the existing AES-256-GCM `secrets` table that executor and managed-IaC credentials already use (§3). Consistency wins.
- **Broad/admin registry credentials for the relay (rejected).** The relay needs read on specific source repos and push on specific destination repos; anything wider (delete, admin, cross-boundary reuse) inflates the compromise ceiling past registry pollution for no capability the pipeline uses.

## Consequences

**Positive**
- The last undecided leg of the promotion path is decided: **every tier now has a named byte transport with a named verify**, and [ADR-0015](0015-cosign-cross-boundary-signing.md) §6's "still-unresolved artifact-bytes transport channel" is resolved.
- **~80% reuse:** skopeo pull-to-OCI-layout, the build-bundle tarball + checksums + cosign-sign, the `.scpbundle` file transport, the install.sh push-by-digest + re-inspect, and the entire M17.4 verify machinery all exist and are proven. The net-new work is the relay orchestration + the credential wiring + the OCI-host allowlist + **vendoring a pinned skopeo into the runtime image and air-gap bundle** (the M17.3 cosign-vendoring pattern — today skopeo is an operator-`PATH` prerequisite, BUILD_AND_TEST §1, and the relay runs it server-side).
- The **#108 residual is closed** by design (§4): both bundle-supplied egress surfaces (blob URLs, OCI registry hosts) sit behind symmetric, fail-closed operator allowlists.
- The commercial tier costs **zero transport code** — writing that down prevents anyone "helpfully" building a commander push path later.
- Defense in depth is explicit: relay compromise ≠ deployment (§3 blast radius; §2 step 7).

**Costs / honesty**
- **Retrans becomes a real component for the first time.** ADR-0004 declared it; this builds it — deployment story (the bundled chart's `federationRole=retrans` currently bundles **zero** backends, M15.4, which remains correct: cosign is already vendored in the SCP image (M17.3 E1) and skopeo becomes vendored as part of this build — binaries in the image, not bundled backends), scheduling of relay runs, and CDS handoff mechanics (drop-directory vs. diode transfer varies per CDS product; the relay's contract ends at "signed tarball out / signed tarball in", the same boundary the `.scpbundle` walk draws today).
- **SCP's credential holdings grow by one class** (§3). Deliberately narrow (artifact-store read/push, per-peer, per-registry, vaulted) and charter-argued, but it is a real expansion and is recorded as such — the honest analogue of the Managed Execution Exception, not smuggled in.
- **`SCP_ARTIFACT_OCI_REGISTRY_HOSTS` unset = fail-closed is a behavior change** for existing operator-loaded deployments whose OCI verifies currently dial the manifest's registry host unguarded. Symmetry and fail-closed-by-default win over silent compatibility; the upgrade requires setting one variable and must be release-noted loudly.
- Large-artifact practicalities (tarball size vs. CDS transfer limits, resumability, partial-set relays) are **not** designed here; the first build takes whole-set tarballs (the install-bundle precedent) and defers chunking until a real CDS constraint demands it.
- The M15.5(c) DoD (BUILD_AND_TEST.md §8) is the proof: a two-registry Testcontainers round-trip — pull from source, validate, tarball, push to destination, receiving M17.4 verify passes — and a tampered/unauthorized artifact **never crosses**; the credential class vaulted + scoped; the OCI-host allowlist enforced.
